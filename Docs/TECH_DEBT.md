# Technical Debt Backlog

This document tracks identified technical debt, bundle optimizations, and code quality improvements to be addressed after the MVP/hackathon phase.

---

## Backlog Items

### TECH-DEBT-001: Optimize Frontend Bundle Size

- **Description**: The production bundle size has increased to ~1033 KB after adding form controls, voice recording assets, image uploads, location capture, and review summaries, which is large for a standard static landing and form layout.
- **Impact**: Slow initial page load times and higher bandwidth usage.
- **Proposed Actions**:
  - Lazy load routes using React's `lazy` and `Suspense`.
  - Split feature bundles into separate chunks using Vite's Rollup configuration.
  - Analyze and optimize imported dependencies (e.g., Lucide React tree-shaking, package-lock audit).
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-002: Refactor Client-side Form Validation to Schema-based Library

- **Description**: The custom input validator hook (`useSubmissionForm.ts`) works well for SPRINT 1 but will scale poorly as additional fields, multimedia inputs, and localized messages are integrated.
- **Impact**: Code complexity and validation maintenance overhead.
- **Proposed Actions**: Migrate validation logic to a schema-based validation library like Zod combined with React Hook Form.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-003: Dynamically Load Configuration Options (Categories/Languages)

- **Description**: Frontend categories and preferred languages lists are currently hardcoded in static client constants.
- **Impact**: Code changes and redeployments are required to add/edit categories.
- **Proposed Actions**: Expose a config endpoint on the backend (e.g., `GET /config`) to fetch available issue categories and languages dynamically.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-004: Route-based Code Splitting

- **Description**: All page components and icons are loaded in a single main JavaScript bundle.
- **Impact**: Increased initial bundle download times and script parsing blocks.
- **Proposed Actions**: 
  - Implement route-based code splitting using `React.lazy()` and `Suspense` for lazy loading route destinations.
  - Specifically isolate split chunks for key workflow pages: Landing, Submission, Voice, Images, and Review.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-005: Distributed Request ID Generation

- **Description**: The current in-memory request ID counter resets when the API process restarts and is not suitable for multi-process or distributed backend node deployments.
- **Impact**: Potential collision of Request IDs in scaled-out production environments.
- **Proposed Actions**: Replace the in-memory counter with a distributed ID generation library, UUIDv7, or rely directly on Firestore document auto-generation schemes.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-006: Use Firestore Server Timestamps

- **Description**: Currently, the serverCreatedAt field is stored as an ISO string generated by the Python datetime library.
- **Impact**: Clock drift between server nodes in distributed deployments can result in inconsistent creation ordering.
- **Proposed Actions**: Migrate serverCreatedAt writes to use Firestore's built-in SERVER_TIMESTAMP sentinel.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-007: Introduce Optimistic Concurrency Control

- **Description**: There is currently no protection against conflicting concurrent updates to a submission document.
- **Impact**: Administrative updates or user revisions could overwrite each other in multi-user environments.
- **Proposed Actions**: Implement optimistic concurrency control via integer document version fields or document last-modified timestamp checks.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-008: Secure Signed Media URLs

- **Description**: Media download URLs are currently generated using public unauthenticated paths.
- **Impact**: Anyone with the link can access the evidence files, exposing private citizen information.
- **Proposed Actions**: Transition Firebase Storage access to short-lived signed URLs generated on-the-fly.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-009: Upload Resilience & Retry Strategy

- **Description**: There is no built-in resilience pattern during network-dependent file uploads to Firebase Storage.
- **Impact**: Client uploads fail abruptly on transient network dropped packages.
- **Proposed Actions**: Introduce a robust retry strategy with exponential backoff on storage client network operations.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-010: Centralized Media Validation Engine

- **Description**: Validation checks for images and voice files (sizes, mime-types, extensions) are handled inline within the upload service.
- **Impact**: Hard to maintain, scale, or share rules if new endpoints or background workers process files.
- **Proposed Actions**: Move media constraints to a central validation engine or utility helper module.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-011: Background Upload Queue

- **Description**: Submissions require active browser focus and network connection to finish uploading.
- **Impact**: Citizens experience upload failures if the browser is closed or connectivity drops mid-upload.
- **Proposed Actions**: Implement a Service Worker based background sync queue or localStorage-based submission retry queue to retry after browser reconnect.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-012: Cancelable Upload Actions

- **Description**: Long-running media uploads cannot be canceled once started.
- **Impact**: Frustrated users on slow connections are forced to refresh the browser to abort.
- **Proposed Actions**: Add AbortController support to apiClient and expose a cancellation method in the useSubmissionWorkflow hook.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-013: Chunked Image Uploads

- **Description**: Large image uploads are transmitted in single HTTP requests.
- **Impact**: High failure rate in rural or congested environments with poor mobile signals.
- **Proposed Actions**: Refactor upload media logic to support chunked file uploads with partial upload state recovery.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-014: Prompt Evaluation Dataset

- **Description**: Prompts are currently validated manually. No fixed benchmark dataset exists for regression testing.
- **Impact**: Prompt changes cannot be measured for accuracy, latency, or consistency regressions before deployment.
- **Proposed Actions**: Build a curated dataset of representative citizen submissions with expected analysis outputs and integrate it into CI via a prompt benchmark runner.
- **Status**: Backlog
- **Priority**: High

### TECH-DEBT-015: Prompt Response Caching

- **Description**: Identical or near-identical submission descriptions invoke the full Gemini pipeline on every analysis trigger.
- **Impact**: Unnecessary API cost and latency for duplicate civic complaints (common in high-volume areas).
- **Proposed Actions**: Introduce a lightweight hash-based cache keyed on normalized submission text to short-circuit AI calls for exact duplicates.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-016: Circuit Breaker for Gemini Gateway

- **Description**: Repeated Gemini failures (timeouts, quota errors) are retried inline with no failure rate tracking.
- **Impact**: A sustained outage causes every analysis request to hang until timeout, degrading the entire API.
- **Proposed Actions**: Implement a circuit breaker pattern in GeminiGateway that opens after N consecutive failures and returns a fast-fail response until the service recovers.
- **Status**: Backlog
- **Priority**: High

### TECH-DEBT-017: Async Background Pipeline Execution

- **Description**: AI analysis runs synchronously inside the HTTP request cycle for `POST /api/v1/analysis/submissions/{id}`.
- **Impact**: Long-running Gemini calls (2–8s each, two stages) block the request thread and risk HTTP gateway timeouts.
- **Proposed Actions**: Move pipeline execution to a background task queue (e.g. Celery + Redis, or Cloud Tasks). The endpoint returns 202 Accepted immediately with a polling or webhook mechanism for completion.
- **Status**: Backlog
- **Priority**: High

### TECH-DEBT-018: Analysis Schema Version Migrations

- **Description**: The `AnalysisResult` schema is written to Firestore without a migration strategy for future field changes.
- **Impact**: Adding or renaming fields in future sprints will leave old documents incompatible with new code.
- **Proposed Actions**: Add a `schemaVersion` field to every analysis document and implement a migration utility that back-fills old documents when the schema evolves.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-019: Embedding-Based Duplicate Detection

- **Description**: Duplicate detection uses heuristic Jaccard overlap on tokenized summary words.
- **Impact**: Low accuracy for semantically identical but lexicographically different submissions (e.g., synonyms).
- **Proposed Actions**: Replace heuristic similarity with semantic embedding similarity (e.g. Vertex AI text embeddings) and vector search.
- **Status**: Backlog
- **Priority**: High

### TECH-DEBT-020: Dynamic Hotspot Thresholds

- **Description**: The hotspot threshold is a fixed constant (settings.HOTSPOT_THRESHOLD = 5).
- **Impact**: Submissions in high-volume regions are constantly flagged as hotspots, while critical issues in low-volume rural areas are missed.
- **Proposed Actions**: Implement historical baseline calculation to dynamically scale the hotspot threshold per ward/locality based on historical volume.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-021: External Policy Configuration

- **Description**: Scoring weights, level thresholds, and category severity scores are defined statically in python policy modules.
- **Impact**: Changing priority formulas or weights requires redeploying the backend codebase.
- **Proposed Actions**: Move policy mappings to an external JSON configuration file (or Firestore document) loaded dynamically at runtime by the Priority Engine.
- **Status**: Backlog
- **Priority**: High

### TECH-DEBT-022: Recommendation Feedback Loop

- **Description**: Recommendations are generated and saved without any downstream validation.
- **Impact**: No insight into whether recommended actions match MP/department actions or resolve issues.
- **Proposed Actions**: Design a recommendation status tracker that links external administrative resolution state back to the generated recommendation.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-023: Priority Score Explanation API

- **Description**: The priority score and level are returned as single values without showing the contributing rule factors.
- **Impact**: Dashboard interfaces lack transparency about why an issue is flagged as CRITICAL or HIGH.
- **Proposed Actions**: Expose the `score_breakdown` audit dict (already generated by PriorityEngine) via the API endpoint so frontend clients can display score breakdowns.
- **Status**: Resolved
- **Priority**: High

### TECH-DEBT-024: Dashboard Pagination

- **Description**: The priority queue in the dashboard returns all prioritized items at once, which could lead to scaling bottlenecks.
- **Impact**: Slower response times as the submission collection grows.
- **Proposed Actions**: Implement server-side pagination with limit and offset query parameters.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-025: Dashboard Caching

- **Description**: High-level summary queries scan and aggregate metrics on every single request or refresh.
- **Impact**: Increased Firestore read costs and higher latency.
- **Proposed Actions**: Implement cache-aside or read-through caching for dashboard summary counters.
- **Status**: Backlog
- **Priority**: High

### TECH-DEBT-026: Trend Analytics

- **Description**: The dashboard only displays structural snapshots of current priorities and regional hotspots.
- **Impact**: Incapacity to observe issue growth or decline rates over time.
- **Proposed Actions**: Build trend analytics and historical time-series datasets.
- **Status**: Backlog
- **Priority**: Medium

### TECH-DEBT-027: GIS Visualization

- **Description**: Hotspot and regional density data is presented as a list.
- **Impact**: Harder spatial navigation for Members of Parliament mapping physical ward boundaries.
- **Proposed Actions**: Integrate mapping SDKs (e.g. Mapbox/Google Maps) to visualize hotspots interactively.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-028: Export Reports

- **Description**: Dashboard portal lacks exporting capabilities for official reports.
- **Impact**: Officials cannot print or archive analytical dashboards easily.
- **Proposed Actions**: Add Excel/PDF export capabilities for Priority Queue and Department Directives.
- **Status**: Backlog
- **Priority**: Low

### TECH-DEBT-029: Real-time Dashboard Updates

- **Description**: The dashboard relies on manual polling or button triggers to reload widget data.
- **Impact**: Delayed notification of high-priority citizen submissions.
- **Proposed Actions**: Add WebSocket connections or Server-Sent Events (SSE) to update queues in real-time.
- **Status**: Backlog
- **Priority**: Low
